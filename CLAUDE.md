# プロジェクト概要

パズルゲームの開発を行うプロジェクト。

マインスイーパーベースの猫救出パズルゲーム。犬（危険）を避けて猫（お宝）を全匹保護するとステージクリア。周回ごとに仲間の猫が増え、お供として連れていける。

# 開発プロセス

## 必須: 計画→実装の順序

**実装前に必ず計画を立てること。** 計画なしに直接コードを書き始めてはならない。

計画フロー:
1. **調査** — 関連する仕様書（`docs/`）とプロトタイプ（`samples/`）を参照して現状を把握する
2. **計画提示** — 何をどのファイルに実装するかを箇条書きで示す（ユーザーの承認を得る）
3. **実装** — 計画に従ってコードを書く
4. **ドキュメント更新** — 変更内容を該当する仕様書に反映する
5. **確認** — 動作・表示・副作用を確認する

小さな修正（typo修正・定数値の調整など）は計画提示を省略してよい。

## ドキュメント更新ルール

コードを変更したら、対応する仕様書も同じコミットで更新すること。

| 変更内容 | 更新対象ドキュメント |
|---------|------------------|
| ゲームルール・バランス変更 | `game-design.md` |
| 画面レイアウト・配色変更 | `ui-spec.md` |
| アニメーション追加・変更 | `animation-spec.md` |
| アセット追加・スプライト変更 | `asset-spec.md` |
| スキルロジック変更 | `skill-logic.md` |
| スコア計算変更 | `scoring-detail.md` |
| 盤面生成アルゴリズム変更 | `board-algorithm.md` |
| state変数の追加・削除・変更 | `state-architecture.md` |
| 機能の実装完了・追加 | `roadmap.md` |
| テキスト追加・変更 | `localization.md` |

仕様書が存在しない新機能は、実装と同時に `docs/` に仕様書を作成する。

## コミットルール

- 1コミット = 1機能または1バグ修正（複数の無関係な変更を混ぜない）
- コミットメッセージはプレフィックスを付ける:
  - `feat:` — 新機能
  - `fix:` — バグ修正
  - `refactor:` — 動作変更なしの整理
  - `docs:` — ドキュメントのみの変更
  - `style:` — CSS・見た目のみの変更

### ブランチ・PRルール

**`main` への直接プッシュは禁止。** すべての変更は作業ブランチ経由でプルリクエストを通してマージすること。

#### ブランチ命名規則

| 種別 | 命名パターン | 例 |
|------|------------|-----|
| 新機能 | `feat/<内容>` | `feat/skill-peek` |
| バグ修正 | `fix/<内容>` | `fix/flood-fill-edge` |
| リファクタ | `refactor/<内容>` | `refactor/board-state` |
| ドキュメント | `docs/<内容>` | `docs/update-asset-spec` |

#### 作業フロー

```
1. main から作業ブランチを作成
   git checkout main && git pull origin main
   git checkout -b feat/<内容>

2. コミットを積む（コミットルール参照）

3. プルリクエストを作成
   - タイトル: コミットメッセージと同じプレフィックス形式
   - 本文: 変更内容・動作確認結果・関連する仕様書を記載

4. レビュー・承認後にマージ
```

#### 禁止操作

- `git push origin main`（直接プッシュ）
- `git push --force` / `git push -f`（強制プッシュ）
- マージ前のセルフマージ（レビューなしでのマージ）

# 禁止事項

## Git・リポジトリ操作

- **`main` に直接プッシュしない** — 必ず作業ブランチを切り、プルリクエストを作成する
- **強制プッシュをしない** — `--force` / `-f` オプションは使用禁止
- **マージ前にセルフマージしない** — レビューなしでのマージは禁止

## 設計・実装

- **仕様書にない機能を勝手に追加しない** — `roadmap.md` の「将来拡張」に記載があっても、指示なしに実装しない
- **プロトタイプの動作を無断で変更しない** — `samples/nyanko-rescue.jsx` はリファレンス。挙動を変えるには仕様書を先に更新する
- **state変数を無断で追加・削除しない** — `state-architecture.md` に記載のない変数を導入する場合は計画段階で明示する
- **アニメーション時間を無断で変更しない** — テンポに直結するため、変更は `animation-spec.md` を先に更新する
- **外部ライブラリを無断で追加しない** — 追加が必要な場合は理由とともにユーザーに確認する

## コード品質

- **コメントで「何をしているか」を説明しない** — コード自体が説明できるよう命名する
- **使われていないコードを残さない** — 削除する場合は完全に削除する（コメントアウトで残さない）
- **型安全性を損なわない** — TypeScript を使う場合は `any` を避ける
- **セキュリティ脆弱性を導入しない** — XSS・インジェクション等に注意する

## UI・演出

- **暴力的表現を使わない**（用語ルール参照）

# 開発ルール

## モバイルファースト
- 長押し非依存（フラグはトグルボタン方式）
- トーストは`position: fixed`（盤面レイアウトを動かさない）
- 演出は`pointerEvents: none`（誤タップ防止）
- シェイクは盤面コンテナのみ（画面酔い防止）
- セルサイズ: 38px固定

## テンポ重視
- ヒント演出: 750ms
- 猫保護演出: 700ms
- 犬噛みつき演出: 1.1s（最も長い、インパクト重視）
- ゲーム操作をブロックする演出は最小限に

## 技術方針
- React 18+、フックベース
- Dockerコンテナー
- 外部ライブラリ最小限
- `localStorage`による永続化
- CSS keyframes主体のアニメーション

## 用語
- 暴力的表現を避ける: 「攻撃」→「保護/サーチ」、「爆破」→「ラッシュ」、「敵」→「野良犬」

# ゲームの要点

- 数字は`犬数/猫数`の2色表示（赤=犬、緑=猫）
- ライフ3、犬を踏むと-1
- 全猫保護でクリア（安全マス全開封は不要）
- 10種類の猫 × 10種類のスキル（1対1対応）
- お供選択式（タイトル画面で1匹選ぶ）
- ゲージ制（最大100、マス開封+5、猫保護+25）
- 4ステージ制（にわ→こうえん→しょうてんがい→はいきょ）
- 周回で犬+1/ステージ（上限30%）
- はいきょクリア→出会い画面→仲間アンロック
- リトライ時は前ステージのlapCats保持
